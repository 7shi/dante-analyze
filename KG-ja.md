# KG ラダー：ステップ 05 → 08

知識グラフは、*理解*と*抽出*を意図的に分離した4つの狭いパスとして構築される。各パスは一つの問いに答え、結合を次のパスに委ねるため、最終的な組み立て（`08-kg`）は純粋なコードのみ――LLM なし――であり、構成によって完全性が保証される。

| ステップ | パス | 答える問い | LLM? |
|----------|------|-----------|------|
| 1 | `05-registry` | 各人物は**何者か**（人物ごとに一つの正規ノード） | あり――型付けのみ |
| 2 | `06-speech` | 各引用スパンで**誰が**話しているか | なし |
| 3 | `07-relations` | **誰が誰に何をするか**（`[n]` タグを引用するエッジ） | あり |
| 4 | `08-kg` | **組み立て**――上記すべてをグラフに結合する | なし |

以下の記述は、各ステップにおけるコミット済み出力の具体例を引用しながら、データが下流に流れる様子を追う。

---

## ステップ 1 ――`05-registry`：場面ごとのラベルを正規ノードに折り畳む

`04-tags` はタグが**場面ごと**に*誰*を指すかを解決した。そのため、同じ人物は場面が異なれば異なる表記を持つ（`Virgilio`、`quel Virgilio`、`'l mio maestro`、……）。レジストリは**すべてのユニットを一度に見る**最初のパスである。場面ごとのラベルを三つの歌全体にわたって一つのノードに折り畳み、正規表記を選び、ノードの**型**を割り当て、表面形とその出現数を添付する。このノード集合が、後続のすべてのステップが `fold_key` で結合する対象となる。

**パイプライン：** `gather（コード）→ fold-merge（コード）→ set-resolve（コード）→ type（LLM、キャッシュ済み）→ render（コード）→ check（コード）`。LLM ステージは型付けのみであり、LLM に問うのは一事だけ――ノードごとのクローズドボキャブラリの型。

- **Fold-merge（純粋なコード）** は 2,922 の表記を決定論的に **2,711 ノード**に折り畳む。正規ラベルは三つの歌全体でグローバルに決定された、グループ内で最も頻出する元の表記であるため、歌をまたぐ人物（Dante、Virgilio、Beatrice）は一つのノードとなり、一度だけ型付けされる。
- **ノード型付け（LLM、キャッシュ済み）** は各非セットノードをクローズドボキャブラリで一度分類する――`individual | generic | class | hypothetical-simile | non-person`――20 件ずつのバッチで。モデルが見て答える具体的なバッチの例：
  ```
  1. Dante = individual
  2. la Fortuna = non-person
  3. angeli = class
  4. Beatrice = individual
  ```
  約 2,550 ノード ÷ 20 ≈ **約 128 回**の呼び出し――場面ごとに型付けするよりはるかに少ない。

**出力例（`inferno.txt`）** ――正規ノード、その表面形、および出現数：

```
## Virgilio
- type: individual
- labels: Virgilio
- surfaces: tu (96), ei (86), io (68), elli (50), Maestro (22), ..., quel Virgilio (1)

## Dante, Virgilio
- type: set
- members: Dante | Virgilio
```

型付けは `04-tags` ではなくここに置かれる。型はタグのプロパティ（場面ごとのローカルな視点）ではなく**ノード**のプロパティ（人物のグローバルな視点）だからである。`04-tags` で型付けすると、同じ人物を何十回も（最大 16,030 タグ行に及ぶ）再分類することになり、場面ごとに型が異なるリスクがある。ここではノードごとに一度（約 2,550 回）型付けすれば済む。

**計測優先の設計。** `measure.py` はプロンプトが存在する前に問題の規模を測定し、**判断ゲート**を生成した。上皮語グループ化ゲート（`epithet nodes/canticle < 150`）で*失敗*したため、v1 は**オプション A** を採用している――上皮語グループ化をスキップし、名前でもセットでもないすべてのノードは `- grouped: no` フラグ付きの独自ノードとして保持する。構造チェックが検証できないマージよりも、フラグ付きシングルトンの方が安全である。

## ステップ 2 ――`06-speech`：各引用スパンで誰が話しているか

正規ノードが手に入ったところで、このパスは最初の KG *エッジ*層を生成する。詩のすべての引用スパンについて**誰が話しているか**を、引用自身の領域内に存在する一人称指示物を読み取り、レジストリの正規ノードに結合することで判定する。**純粋なコード、LLM なし**――作業は幾何学（どのタグがどの引用の内側にあるか）と結合のみである。

スパンごとの処理：

1. **指示物の収集** ――各タグの `04-tags` ラベルを `norm_label` 化してから**レジストリを通じて正規化する**（`fold_key(label) → 正規見出し`）。
2. **スパンごとの帰属** ――スパンの*固有領域*（スパン内にあるがその子スパンの内側にはない領域。ネストした引用は子に帰属する）に `(line, col)` が含まれる正規指示物を、一人称表面形（`io`/`i'` は強、`mi`/`me` は弱、複数形）でバケット分けして収集する。
3. **話者 / シグナル** ――唯一性テストの前に正規化することで、同一人物の二つの表記が一人の話者に折り畳まれ、`multi` にならない：
   - 唯一の強一人称（`io`） → その話者、`signal: strong`；
   - 複数の異なる強指示物 → `(unattributed)`、フラグ `multi(<a>;<b>;…)`；
   - それ以外で唯一の弱 → その話者、`signal: weak`；
   - それ以外 → `(unattributed)`。

**出力例（`inferno/01.txt`）：**

```
# Canto 01 — The Dark Wood and the Encounter with Virgil
- 1:65 lines 65-65 | speaker: Dante | signal: weak | flags: -
- 1:67 lines 67-78 | speaker: Virgilio | signal: strong | flags: cross-scene
- 1:79 lines 79-80 | speaker: (unattributed) | signal: none | flags: -
```

**計測済みカバレッジ**（`05-registry/measure.py` がこの計算の規模を測定済み）：**1,222 の引用スパン**、レジストリ結合前にコードのみで（列を考慮して）解決：strong-unique 395 / multi-strong 28 / weak-only 90 / plural-only 68 / none 641。レジストリ正規化後のコミット済み数値は**strong 398**（≥ 395――正規化はマージするだけで分割しない）、**weak 90**、**none 734** に一致する。列は重要である――一行の引用はタグ位置が行番号だけでなくソース列を持つため、正確に解決される。

カバレッジは**計測データであり、目標値ではない**――v1 ではほとんどのスパンが `(unattributed)` であるが、これはバグではなく想定通りである。

## ステップ 3 ――`07-relations`：場面ごとの主語―述語―目的語エッジ

このパスは残りの KG の問い――**誰が誰に何をするか**――に、`04-tags` のタグ番号を引用する行指向エッジとして答える。ステップ 4 がレジストリノードに機械的に結合できるのはそのためである。`04-tags` と同様に**解釈に依存する**（CoT オン）、コミット済みの読みに束縛され、構造チェックと会話内リトライによってゲートされる。

**クローズド述語ボキャブラリを用いる理由（計測優先）。** `measure.py` は読みの `-s` 動詞（読みは三単現で書かれており、英語の三単現動詞はすべて `-s` で終わる："Virgil **explains**…"、"Dante **asks**…"）を収集し、`-s` で終わる二つの非述語クラスを差し引いた：複数名詞（`souls`、`spirits`）と**メタ言説**動詞（`describes` 250 回、`explains` 247 回、`continues` 168 回）。物語内の残りは **31 の正規述語**（`CLOSED_VOCAB`）に折り畳まれ、≤40 の扱いやすさゲートをクリアした――**一つのクローズドリスト、グループ化パスなし**（上皮語ゲートが*失敗した*レジストリとは対照的）。

`CLOSED_VOCAB` は**唯一の真実のソース**：`relations.py` は `from measure import CLOSED_VOCAB` し、プロンプトのメニューと構造チェックの両方が `set(CLOSED_VOCAB)` を読む――第二のコピーは存在しない。

**場面ごとに一つの会話で二ターン：**

1. ターン 1 ――`number_scene` タグ付き場面の上に、コミット済みの読みをアシスタントターンとして再生する。
2. ターン 2 ――そのタグ付き場面に対してエッジリストを要求する。

引用された `[n]` は `04-tags` が解決した**同一の**場面ごとタグ番号である――このパスは番号を振り直さないため、ステップ 4 の結合が完全になる。

**関係行の文法：**

```
- [<subj>] <predicate> [<obj>] | frame: <literal|simile|prophecy|reported> | lines <a>-<b>
```

**出力例（`inferno/01.txt`）** ――四つのフレームすべてが登場：

```
## Scene 22-30: The Swimmer Simile
- [3] compares [1] | frame: simile | lines 22-26

## Scene 31-36: The Appearance of the Leopard
- [1] chases [2] | frame: literal | lines 34-34
- [3] chases [2] | frame: literal | lines 35-35

## Scene 67-75: Virgil's Introduction
- [13] praises [14] | frame: reported | lines 73-74

## Scene 112-120: The Plan for the Journey
- [6] guides [4] | frame: prophecy | lines 113-113
```

**`frame` は後付けではなく構造的なもの**：`literal` = 直接語られた出来事；`reported` = 登場人物が語る内容；`prophecy` = 予告された未来の出来事；`simile` = 比喩の比較対象側。意図的に**`says-that` メタエッジは存在しない**――reported/prophecy/simile の命題は対応するフレームを持つ*内容*エッジとして出力され、*誰がそれを主張したか*はエッジの行範囲を `06-speech` の話者に結合することで下流で回収される――実際の発話行為（`[a] asks [b]`）はそれ自体が通常の `literal` エッジとなる。

**チェックは構造のみを検証する** ――引用されたすべての `[n]` が場面のタグセットに存在するか、すべての述語が `CLOSED_VOCAB ∪ {relates-to}` に含まれるか、すべてのフレームが四値セットに含まれるか、すべての `lines a-b` が場面内に収まるか。エッジが*正しい*関係であるかどうかは解釈であり、生成されたものとして出荷される。`inferno/01.txt` には二つの結果が見られ、データとして受け入れられている：Scene 1-12 は `[1] meets [2]`（両タグが Dante――同一レジストリノードフィルタはステップ 4 に属し、そこでノード同一性が解決される）を出力し、Scene 100-105（Veltro）は散文が予言するにもかかわらず `prophecy` エッジを生成しない。

## ステップ 4 ――`08-kg`：グラフを組み立てる（純粋なコード、LLM なし）

最後のステップは、コミット済みの上流出力を組み立てられたグラフに**結合する**。すべての入力は `dante_analyze` の `load_*` API を通して読まれる。ここでテキストは再解釈されない。歌集ごと、歌ごと、`load_relations(canticle, canto)` に対して：

- **各エッジ端をノードに解決する。** エッジの `lines a-b` はちょうど一つの `04-tags` 場面内に収まる（場面は歌を分割する）。その場面を見つけ、引用された `[subj]`/`[obj]` を `load_tags → 名前` でマップし、名前をレジストリで正規化する（`raw_to_canonical` → `fold_key` → 正規見出し）→ グラフ**ノード**。
- **主張者を回収する。** `reported`/`prophecy`/`simile` エッジの場合、主張者はエッジの行範囲を含む**最も内側の** `06-speech` 引用スパンの話者である。`literal` エッジは語り → `null`；`(unattributed)` のコンテナも `null` を返す。
- **スピーチエッジをマージする**（話者 → 引用スパン）と関係エッジを並べて。

**出力は歌集ごと、JSONL** ――`nodes.jsonl`、`edges.jsonl`、`speech_edges.jsonl`。

`nodes.jsonl` ――`id` はレジストリの正規見出し；表面形・ラベルは省略（グラフはノードを `id` で参照する）：
```jsonl
{"id": "Dante", "type": "individual", "members": null}
{"id": "Virgilio", "type": "individual", "members": null}
{"id": "Dante, Virgilio", "type": "set", "members": ["Dante", "Virgilio"]}
```

`edges.jsonl` ――`[n]` タグはノードに解決され、主張フレームの場合は主張者が回収される：
```jsonl
{"canto": 1, "scene": [49, 60], "subj": {"tag": 2, "name": "la lupa", "node": "la lupa"}, "predicate": "punishes", "obj": {"tag": 3, "name": "Dante", "node": "Dante"}, "frame": "literal", "lines": [52, 52], "asserter": null}
{"canto": 1, "scene": [67, 75], "subj": {"tag": 8, "name": "Virgilio", "node": "Virgilio"}, "predicate": "relates-to", "obj": {"tag": 9, "name": "Augusto", "node": "Augusto"}, "frame": "reported", "lines": [71, 71], "asserter": "Virgilio"}
```

二番目のエッジの `asserter: "Virgilio"` は、その `lines [71, 71]` をステップ 2 で示した `06-speech` スパン `1:67 lines 67-78 | speaker: Virgilio` に結合することで回収される――`07-relations` が意図的に先送りにした結合である。

`speech_edges.jsonl` ――`06-speech` スパンをそのまま運搬：
```jsonl
{"canto": 1, "quote_id": "1:67", "lines": [67, 78], "speaker": "Virgilio", "signal": "strong", "flags": ["cross-scene"]}
```

**構造チェック。** 幾何学は致命的（すべてのエッジがちょうど一つの場面内、引用されたすべての `[n]` が存在する）；名前→ノード解決は集計のみで致命的ではない（未解決の端は `node: null` として書き込まれ、カウントされる）。これまでのすべての実行において、未解決の端はレジストリが正しく削除した `(unknown)` ラベルである。

**計測結果：**

| 歌集 | ノード | エッジ | スピーチエッジ |
|------|-------:|-------:|---------------:|
| inferno    | 1045  | 2142  |          500 |
| purgatorio |  936  | 2010  |          495 |
| paradiso   | 1050  | 1604  |          227 |

幾何学：全 100 歌にわたり**失敗 0 件**。名前→ノード：**未解決エッジ端合計 18 件**（inferno 2、purgatorio 10、paradiso 6）、すべて `(unknown)` ラベル。

## ラダーがこのように分割されている理由

各パスは**一種類の作業**を行い、**一つのチェック**を持つ：

- `04-tags` は*誰か*を答える（`n. Name`）；`05-registry` は*どんな種類の指示物か*（型）を答える。これらは直交する問いであり、型はタグのプロパティ（場面ごと）ではなくノードのプロパティ（グローバル）である――`04-tags` に組み込むと、同じ人物を何十回も再型付けし、場面ごとに異なる型が生じるリスクがある。
- `06-speech` は幾何学と結合によって*誰が話しているか*を答える――純粋なコードなので、結合先のレジストリと矛盾しない。
- `07-relations` は*誰が誰に何をするか*を答え、`[n]` タグを引用するだけで解決しない。そのため解釈パスはノード同一性を知る必要がない。
- `08-kg` は先送りにされた結合である：上流チェック（タグ番号が整合している、引用されたすべての `[n]` が存在する、すべてのエッジの行範囲が一つの場面に収まる）がまさに、コードが各エッジ端をノードに機械的に解決できる根拠であるため、構成によって完全性が保証される。

計測優先の規律（`05-registry` と `07-relations` の両方に `measure.py`）は、プロンプトを固める前に**完全な**出力で LLM 残差の規模を測定し、残差が扱いやすいかどうかを示す判断ゲートを生成する。レジストリの上皮語ゲートは**失敗した** → v1 はフラグ付きシングルトン（`grouped: no`）を採用；関係の述語ゲートは**通過した**（31 ≤ 40） → 一つのクローズドリスト、グループ化パスなし。

精度は**方法を変えることで向上させ、個別修正は行わない**：検証不可能なマージよりフラグ付きシングルトン；手で修正するのではなく計測として数える `node: null` 残差。
