# KG ラダー：ステップ 05 → 08

知識グラフは、*理解*と*抽出*を意図的に分離した4つの狭いパスとして構築される。各パスは一つの問いに答え、結合を次のステップに委ねる。そのため最終的なアセンブリ（`08-kg`）は純粋なコードであり — LLM を使わず — 構造上完全である。

| ステップ | パス | 答える問い | LLM? |
|---------|------|-----------|------|
| 1 | `05-registry` | 各ラベル群の**型**は何か（表記正規化 + 型付け）？ | なし — 純粋なコード（型付けは前段 `04-tags/node_types.py`） |
| 2 | `06-speech` | 各引用スパンを**誰が**話しているか？ | なし |
| 3 | `07-relations` | **誰が誰に何をするか**（`[n]` タグを参照したエッジ）？ | あり |
| 4 | `08-kg` | **アセンブル** — 上記すべてをグラフに結合する。 | なし |

以下の説明は、各ステップのコミット済み出力から具体例を引用しながら、データの流れを追う。

---

## ステップ 1 — `05-registry`：ラベルをノードに正規化し、型付けする

`04-tags` は各タグをシーン**ごとに**命名する — 読解に基づいて、すべてのマーク（`[tu]`、`{Virgilio}`、`'l mio maestro`）が何を指すかを解決する — しかしタグ間のクロスリンクは行わない：「このタグとあのタグは同じ人物」とは決して言わないため、同一人物が異なるシーンで異なるラベルを持つ可能性がある（`il Navarrese`、`Navarrese`、`Lo Navarrese` など）。レジストリは**すべてのユニットを同時に**参照する最初のパスであり、表記の揺れを正規化し（大文字小文字の統一 + 冠詞の除去 — `il Navarrese` と `Navarrese` は同一ノードになる）、3つのカンティカ全体で**グローバルに**正準表記を選択し、各ノードの**型**（前段の型付けキャッシュ `04-tags/types.txt` から読み込む）を付与し、マークアップが持つ表層形（代名詞、換喩）を収集する。このノード集合が、後続のすべてのステップが `fold_key` で結合する対象となる。

レジストリが**行わないこと**：クロスタグの照応解決 — 同一人物の本当に異なるラベルを統合すること（裸の `Guido` = `Guido da Montefeltro`、`Maestro Adamo` = `Mastro Adamo`）— は v1 では試みない。識別ギャップとグラフへの影響については `KG-PROBLEM.md` を参照。

**パイプライン：** `gather（コード）→ fold-merge（コード）→ set-resolve（コード）→ read types（コード）→ render（コード）→ check（コード）`。レジストリのビルド自体は**モデルを呼ばない純粋なコード**であり、ノードの型は前段の LLM ステップ `04-tags/node_types.py` が生成したキャッシュ `04-tags/types.txt` から読み込むだけである。

- **Fold-merge（純粋なコード）** は表記の揺れを正規化する：大文字小文字の統一 + 冠詞の除去（`la Fortuna`/`Fortuna`、`i Malebranche`/`Malebranche`、`il Navarrese`/`Navarrese`）。**2,923 の個別ラベル → 2,712 ノード**へ圧縮 — 149 のグループが 360 のラベルを統合し、すべて大文字/冠詞の変異体。統合は表記のみ：内容トークンが異なるラベル（`Maestro Adamo` vs `Mastro Adamo`、裸の `Guido` vs `Guido da Montefeltro`）は別ノードのままとなる。正準ラベルは3つのカンティカ全体でグローバルに決定された最頻出の元の表記であるため、カンティカ横断の人物（ダンテ、ヴィルジリオ、ベアトリーチェ）は一つのノードとなり、一度だけ型付けされる。
- **ノード型付け** はキャッシュ `04-tags/types.txt` から読み込まれる。これは前段の LLM ステップ `04-tags/node_types.py`（レジストリのビルド自体はコードのみ）が、各非集合ノードをクローズドボキャブラリー — `individual | generic | class | hypothetical-simile | non-person` — で一度だけ分類して生成したものである（バッチサイズ 20）。モデルが受け取り回答する具体的なバッチ：
  ```
  1. Dante = individual
  2. la Fortuna = non-person
  3. angeli = class
  4. Beatrice = individual
  ```
  ~2,550 ノード ÷ 20 ≈ **~128 コール** — シーンごとに型付けするより大幅に少ない。

**出力例（`inferno.txt`）** — 正準ノード、ラベル、表層収集：

```
## Virgilio
- type: individual
- labels: Virgilio
- surfaces: tu (96), ei (86), io (68), elli (50), Maestro (22), ..., quel Virgilio (1)

## Dante, Virgilio
- type: set
- members: Dante | Virgilio
```

`labels` のエントリが一つなのは、`04-tags` がシーンをまたいで一貫して "Virgilio" という名前を使用しているため（読解全体で名前によって識別される）；`surfaces` はマークアップが持つ**テキスト形式**（代名詞、換喩、名前そのもの）の収集であり、レジストリが統合したノードではない。`04-tags` のラベルが表記的に揺れた人物（`il Navarrese` / `Navarrese` / `Lo Navarrese`）は fold-merge で一つのノードに集約され、ラベルが本当に異なる言葉の人物（`Maestro Adamo` vs `Mastro Adamo`）は別ノードのままとなる（`KG-PROBLEM.md` を参照）。

型付けは**ノードレベル**のパス（`04-tags/node_types.py`、このビルドの前に実行）であり、シーンごとの `tags.py` に組み込まれてはいない。理由：型は**ノード**のプロパティ（人物のグローバルな視点）であり、タグのプロパティ（シーンごと）ではないからだ。シーンごとに型付けすると、同一人物が何十回も（最大 16,030 タグ行で）再分類され、シーンによって型が異なるリスクがある。一方ノードレベルのパスではノードごとに一度だけ型付けされる（~2,550）。これが `04-tags` ディレクトリに置かれるのは、コミット済みの `04-tags` ラベルだけに依存し、レジストリ（および `types.txt` を読む coreference）より前に走るため — ビルドを一直線に保つ。

**計測優先設計。** `measure.py` は任意のプロンプトが存在する前に問題のサイズを計測し、**決定ゲート**を生成した。どちらも失敗した：近似重複ゲート（`長い形式を持つベース人物 < 50` → 346）と換喩ゲート（`換喩ノード/カンティカ < 150` → 285-330）。v1 は **オプション A** を採用：換喩グループ化はスキップされ、すべての非名前・非集合ノードは `grouped: no` とフラグされた独立したノードを保持する。構造チェックで検証できないマージよりも、フラグ付きシングルトンの方が安全だ。近似重複がほぼ偽陽性である理由と識別ギャップのグラフへの意味については `KG-PROBLEM.md` を参照。

## ステップ 2 — `06-speech`：各引用スパンを誰が話しているか

正準ノードを手にして、このパスは最初の KG *エッジ*レイヤーを生成する：詩中のすべての引用スパンについて**誰が話しているか**を、引用の範囲内に落ちる一人称照応をレジストリの正準ノードに結合することで決定する。**純粋なコード、LLM なし** — 作業は幾何学（どのタグがどの引用の中にあるか）と結合である。

スパンごとの処理：

1. **照応の収集** — 各タグの `04-tags` ラベルを `norm_label` してから**レジストリで正準化**する（`fold_key(label) → 正準見出し`）。
2. **スパンごとの帰属** — `(line, col)` がスパン固有の領域（スパン内かつ子スパンの外にある — ネストされた引用は子に属する）に含まれる正準照応を収集し、一人称表層（`io`/`i'` が強、`mi`/`me` が弱、複数形）でバケット分け。
3. **話者 / シグナル** — 一意性テストの前に正準化するため、同一人物の二つの表記が `multi` ではなく一人の話者に収束する：
   - 固有の強い一人称（`io`）→ その話者、`signal: strong`；
   - 複数の異なる強い照応 → `(unattributed)`、フラグ `multi(<a>;<b>;…)`；
   - それ以外で固有の弱い照応 → その話者、`signal: weak`；
   - それ以外 → `(unattributed)`。

**出力例（`inferno/01.txt`）：**

```
# Canto 01 — The Dark Wood and the Encounter with Virgil
- 1:65 lines 65-65 | speaker: Dante | signal: weak | flags: -
- 1:67 lines 67-78 | speaker: Virgilio | signal: strong | flags: cross-scene
- 1:79 lines 79-80 | speaker: (unattributed) | signal: none | flags: -
```

**計測カバレッジ**（`05-registry/measure.py` がこの計算のサイズを測定）：**1,222 の引用スパン**、コードのみで（列を考慮し、レジストリ結合前に）strong-unique 395 / multi-strong 28 / weak-only 90 / plural-only 68 / none 641 と解決。レジストリ正準化後のコミット済み数値は **strong 398**（≥ 395 — 正準化は統合のみで分割しない）、**weak 90**、**none 734**。列は重要：単一行の引用はタグ位置がソース列を持つため正確に解決される。

カバレッジは**測定データであり、目標ではない** — v1 ではほとんどのスパンが `(unattributed)` となるが、これはバグではなく想定内である。

## ステップ 3 — `07-relations`：シーンごとの主語-述語-目的語エッジ

このパスは残りの KG の問いに答える — **誰が誰に何をするか** — `04-tags` のタグ番号を参照した行指向のエッジとして表現し、ステップ 4 がそれらをレジストリノードに機械的に結合できるようにする。`04-tags` と同様に**解釈に束縛**されており（CoT オン）、コミット済みの読解に拘束され、インコンバーセーションリトライを伴う構造チェックでゲートされる。

**クローズド述語ボキャブラリーを選ぶ理由（計測優先）。** `measure.py` は読解の `-s` 動詞を収集した（すべての英語三人称単数現在動詞は `-s` で終わり、読解はそのテンスで書かれている："Virgil **explains**…"、"Dante **asks**…"）、次に `-s` で終わる二つの非述語クラスを除いた：複数名詞（`souls`、`spirits`）と**メタ言説**動詞（`describes` 250×、`explains` 247×、`continues` 168×）。物語内の残りは**31 の正準述語**（`CLOSED_VOCAB`）に収束し、≤40 の扱いやすさゲートをクリアした — **一つのクローズドリスト、グループ化パスなし**（失敗したレジストリの換喩ゲートとは対照的）。

`CLOSED_VOCAB` は**単一の正規情報源**：`relations.py` は `from measure import CLOSED_VOCAB` を実行し、プロンプトメニューと構造チェックの両方が `set(CLOSED_VOCAB)` を読む — 二つ目のコピーは存在しない。

**シーンごとに一つの会話で二ターン：**

1. ターン 1 — `number_scene` タグ付きシーンに対するアシスタントターンとして、コミット済みの読解を再生する。
2. ターン 2 — そのタグ付きシーンに対するエッジリストを要求する。

参照される `[n]` は `04-tags` が解決した**同一の**シーンごとのタグ番号 — このパスは番号を振り直さないため、ステップ 4 の結合が完全になる。

**リレーション行の文法：**

```
- [<subj>] <predicate> [<obj>] | frame: <literal|simile|prophecy|reported> | lines <a>-<b>
```

**出力例（`inferno/01.txt`）** — 4つのフレームすべてが登場：

```
## Scene 22-30: The Swimmer Simile
- [3] compares [1] | frame: simile | lines 22-26

## Scene 31-36: The Appearance of the Leopard
- [1] chases [2] | frame: literal | lines 34-34
- [3] chases [4] | frame: literal | lines 35-36

## Scene 67-75: Virgil's Introduction
- [13] tells [14] | frame: reported | lines 73-73

## Scene 112-120: The Plan for the Journey
- [6] guides [4] | frame: prophecy | lines 113-113
```

**`frame` は構造的であり、事後付けではない**：`literal` = 直接語られた出来事；`reported` = 登場人物が語る内容；`prophecy` = 予言された将来の出来事；`simile` = 比較の比喩的な側面。意図的に**`says-that` メタエッジはない**：reported/prophecy/simile の命題は対応するフレームを持つ*内容*エッジとして出力され、*誰がそれを主張したか*はエッジの行範囲を `06-speech` の話者に結合することで下流で復元される — 文字通りの発話行為（`[a] asks [b]`）はそれ自体が通常の `literal` エッジである。

**チェックは構造のみを証明する** — 引用される `[n]` がシーンのタグ集合に存在すること、すべての述語が `CLOSED_VOCAB ∪ {relates-to}` に属すること、すべてのフレームが4値集合にあること、すべての `lines a-b` がシーン内にあること。エッジが*正しい*関係であるかどうかは解釈であり、生成されたものとして出荷される。`inferno/01.txt` に二つの結果が見られ、データとして受け入れられる：シーン 1-12 は `[1] meets [2]` を出力するが、両方のタグがダンテである（タグ*番号*が異なる自己関係 — 「同じレジストリノード」フィルターはノード同一性が解決されるステップ 4 に属する）；シーン 100-105（ヴェルトロ）は散文が予言するにもかかわらず `prophecy` エッジを生成しない。

## ステップ 4 — `08-kg`：グラフのアセンブル（純粋なコード、LLM なし）

最後のステップは、コミット済みの上流出力をアセンブルされたグラフに**結合**する。すべての入力は `dante_analyze` の `load_*` API を通じて読み込まれ、テキストの再解釈はここでは行わない。カンティカごと、カントごとに `load_relations(canticle, canto)` を処理：

- **各エッジの端をノードに解決する。** エッジの `lines a-b` は `04-tags` のシーン（シーンはカントを分割する）の一つに正確に収まるため、そのシーンを見つけ、参照された `[subj]`/`[obj]` を `load_tags → 名前` にマップし、レジストリで名前を正準化する（`raw_to_canonical` → `fold_key` → 正準見出し）→ グラフ**ノード**。
- **主張者を復元する。** `reported`/`prophecy`/`simile` エッジについては、主張者はエッジの行範囲を含む最も内側の `06-speech` 引用スパンの話者である。`literal` エッジは語りとなる → `null`；`(unattributed)` コンテナも `null` を返す。
- **スピーチエッジを統合する**（話者 → 引用スパン）リレーションエッジとともに。

**出力はカンティカごと、JSONL** — `nodes.jsonl`、`edges.jsonl`、`speech_edges.jsonl`。

`nodes.jsonl` — `id` はレジストリの正準見出し；表層/ラベルは削除される（グラフは `id` でノードを参照する）：
```jsonl
{"id": "Dante", "type": "individual", "members": null}
{"id": "Virgilio", "type": "individual", "members": null}
{"id": "Dante, Virgilio", "type": "set", "members": ["Dante", "Virgilio"]}
```

`edges.jsonl` — `[n]` タグがノードに解決され、主張フレームの主張者が復元される：
```jsonl
{"canto": 1, "scene": [49, 60], "subj": {"tag": 2, "name": "la lupa", "node": "la lupa"}, "predicate": "punishes", "obj": {"tag": 3, "name": "Dante", "node": "Dante"}, "frame": "literal", "lines": [52, 52], "asserter": null}
{"canto": 1, "scene": [67, 75], "subj": {"tag": 8, "name": "Virgilio", "node": "Virgilio"}, "predicate": "relates-to", "obj": {"tag": 9, "name": "Augusto", "node": "Augusto"}, "frame": "reported", "lines": [71, 71], "asserter": "Virgilio"}
```

二番目のエッジの `asserter: "Virgilio"` は、その `lines [71, 71]` をステップ 2 で示した `06-speech` スパン `1:67 lines 67-78 | speaker: Virgilio` に結合することで復元される — `07-relations` が意図的に先送りした結合である。

`speech_edges.jsonl` — `06-speech` スパンをそのまま転記：
```jsonl
{"canto": 1, "quote_id": "1:67", "lines": [67, 78], "speaker": "Virgilio", "signal": "strong", "flags": ["cross-scene"]}
```

**構造チェック。** 幾何学は致命的（すべてのエッジが正確に一つのシーン内にあり、引用されたすべての `[n]` が存在する）；名前→ノードの解決は集計されるが致命的ではない（未解決の端は `node: null` として書き込まれ、カウントされる）。これまでのすべての実行で、各未解決の端はレジストリが正しく削除した `(unknown)` ラベルである。

**計測結果：**

| カンティカ   | ノード | エッジ | speech_edges |
|------------|------:|------:|-------------:|
| inferno    | 1045  | 2142  |          500 |
| purgatorio |  936  | 2010  |          495 |
| paradiso   | 1050  | 1604  |          227 |

幾何学：全 100 カントで **失敗 0**。名前→ノード：**未解決エッジ端 18 件**（inferno 2、purgatorio 10、paradiso 6）、すべて `(unknown)` ラベル。

## なぜラダーがこのように分割されているか

各パスは**一種類の作業**を行い、**一つのチェック**を持つ：

- `04-tags/tags.py` は各タグをシーンごとに命名する（タグごとの解決、読解に束縛）；型付け（`04-tags/node_types.py`）はノードを分類し、`05-registry` はラベルの表記をノードに正規化する（純粋なコード）。これらは別のステップである：シーンごとのパスはシーンをまたいで正規化できず、型はノードのプロパティ（グローバル）であってタグのプロパティ（シーンごと）ではないからだ — 型付けをシーンごとの `tags.py` に組み込むと、同一人物が何十回も再型付けされ、シーンごとに型が異なるリスクがあるため、ノードレベルのパスとしている。クロスタグの照応解決（同一人物の本当に異なるラベルをリンクすること）は v1 では試みない；`KG-PROBLEM.md` を参照。
- `06-speech` は*誰が話しているか*を幾何学 + 結合で答える — 純粋なコードであるため、結合するレジストリと矛盾できない。
- `07-relations` は*誰が誰に何をするか*に答え、`[n]` タグを参照するが解決しない。そのため解釈パスはノード同一性を知る必要がない。
- `08-kg` は先送りされた結合であり：構造上完全。なぜなら上流のチェック（タグ番号が整合している、引用されたすべての `[n]` が存在する、すべてのエッジの行範囲が一つのシーン内にある）が、コードが各エッジの端を機械的にノードに解決できる正確な条件だからだ。

計測優先の規律（`05-registry` と `07-relations` の両方の `measure.py`）は、プロンプトを確定する前に**完全な**出力に対して LLM の残余をサイズ測定し、残余が扱いやすいかどうかを示す決定ゲートを生成する。レジストリの換喩ゲートと近似重複ゲートは両方**失敗した** → v1 はフラグ付きシングルトン（`grouped: no`）を出荷；リレーションの述語ゲートは**合格した**（31 ≤ 40）→ 一つのクローズドリスト、グループ化パスなし。

精度は**方法を変えることで向上させる。アイテムごとのパッチは行わない**：検証できないマージよりもフラグ付きシングルトン；手動修正ではなく計測として数えられる `node: null` の残余。
