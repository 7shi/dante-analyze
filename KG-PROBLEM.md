# Node identity and its downstream impact

> **Status.** The identity-resolution *mechanism* is in place. Fix 1 (deterministic alias merge,
> `05-registry/aliases.txt`) and Fix 2 (per-tag context-aware coreference overlay,
> `04-tags/coref.txt`, generated by `05-registry/coreference.py`) are both implemented; how each
> works lives in `05-registry/README.md` ("Identity resolution beyond fold_key"). What remains is
> **operational, not architectural**: actually running Fix 2 over the corpus, reviewing its
> proposals, and propagating the resulting node-set change through the downstream passes. This
> document is now about that propagation — **the downstream impact of a node-set change** — which is
> the part with no automatic safety net.

## Why node identity matters

Two requirements drive the knowledge graph:

1. **Accurate character listing** — one node per actual figure, no duplicates, no phantom entries.
2. **Person-centric dynamics extraction** — querying a figure returns *all* of its edges, nothing
   silently missing.

Both depend on **node identity**. If one person is split across two nodes (bare `Guido` vs.
`Guido da Montefeltro`), the listing over-counts and a query on either node returns a partial view.
`04-tags` resolves each tag *per scene* and never cross-links tags across scenes; `fold_key` in the
registry merges only cosmetic drift (case, articles, elision). The two identity layers added on top
(Fix 1, Fix 2) close the rest — see `05-registry/README.md`.

The registry over-counts `individual` nodes by roughly **2×** (1,026 nodes vs. ~300–500 real named
figures) from three sources, all of which must be resolved at **node-construction time** (no
downstream filter can remove them): fragmentation (same figure, many labels), one-off periphrastic
singletons, and demonstrative labels (`quel X`, `colui che …`) that name a different person in each
scene. Fix 2 targets the first; the latter two remain open (see "Remaining identity work").

---

## The structural fact that shapes everything

> Steps `06`–`08` and `11`/`13` join to the registry by `fold_key → canonical` (`raw_to_canonical`).
> This map is **global**: it cannot route one surface form to two different nodes. So per-tag
> coreference **cannot** be expressed as a registry edit — it must make the *label itself*
> identity-first, applied at the one place every consumer reads tags (`load_tags`). That is exactly
> what the Fix 2 overlay does. Once labels are identity-first, the existing join folds each tag onto
> the right node with **no change to the join or to 06/08/11**.

The flip side: a correct node set is necessary but the propagation is **not free**. The passes that
consume identity are spread across the pipeline, some cache their LLM output, and there is **no
granular invalidation** — a node-set change does not automatically re-run the passes that depended on
the old identities.

## Downstream impact of a node-set change

How a Fix-2 correction (e.g. bare `Guido` → `Guido da Montefeltro` in one scene) reaches each pass:

| Pass | Registry dependency | LLM / code | Auto-propagates on rerun? | Cache to clear |
|------|--------------------|-----------|--------------------------|----------------|
| 06-speech | direct (`raw_to_canonical`, `load_tags`) | code | **yes** (no cache) | — |
| 07-relations | none (interpretation labels; joined at 08) | LLM | n/a structurally | `done_scene_ends` |
| 08-kg | direct (`raw_to_canonical`, `load_tags`) | code | **yes** (no cache) | — |
| 09-location | none (source text only) | LLM | unaffected | `done_scene_ends` |
| 10-topography | none (09 surfaces only) | LLM | unaffected | `<canticle>.clusters.txt` |
| 11-presence | direct (type filter, set expand, `load_tags`) | LLM | partial | `done_scene_ends` |
| 12-addressee | transitive (via 11 roster) | LLM / code | partial | `done_scene_ends` |
| 13-cohort | direct (type filter) | LLM / code | partial | `done_scene_ends` |
| 14-lock | none (pure join of upstream) | code | **yes** | — |
| 15-digest | none (lock vocabulary) | LLM | follows 14 | `complete_scene_ends` |

### Three findings this table encodes

1. **Code passes propagate for free; LLM passes do not.** 06-speech, 08-kg, and 14-lock are pure
   code with no resume cache: rerun them and they reflect the new node set immediately. Every pass
   that calls the model caches finished scenes (`done_scene_ends` / `clusters.txt` /
   `complete_scene_ends`) and will **silently skip** the scenes whose identities changed. Those need
   a **manual cache clear** for the affected cantos before rerun.

2. **The residual gap is 07-relations.** It emits subject/object labels from its own reading of the
   text, *not* through `load_tags`, so the Fix-2 overlay never touches them. They are canonicalized
   only at the 08-kg join via `raw_to_canonical`. If a relation label stays under-specified (`Guido`)
   while the merged node now carries `Guido da Montefeltro`, the join can miss — the edge attaches to
   a stale or absent node. Closing this means either feeding 07 the corrected labels or applying the
   overlay to its output too. **Not done; this is the next concrete item.**

3. **11/13 are only partially affected, and not always for the better.** A bare-name merge renames a
   roster/cohort entry, which changes the figure the presence/cohort LLM was asked about — so those
   scenes genuinely need regeneration. But 13-cohort filters to `class`/`generic` nodes, and most
   Fix-2 merges are `individual`, so its candidate set rarely moves.

### Recommended rebuild order

After regenerating the registry (`make -C 05-registry`):

```
06-speech, 08-kg        (code, no cache — just rerun)
07-relations            (only if re-deriving relation labels; clear done_scene_ends for changed cantos)
11-presence             (clear done_scene_ends for changed cantos, then rerun)
12-addressee, 13-cohort (depend on 11 / registry types; clear caches for changed cantos)
14-lock                 (code, no cache — rerun)
15-digest               (clear complete_scene_ends for changed cantos, then rerun)
```

09-location and 10-topography have **no registry dependency** and never need rebuilding for an
identity change. The cost of a node-set change is therefore concentrated in the LLM passes that cache
(07, 11, 12, 13, 15) — and only for the **cantos a correction actually touched**, which the overlay
makes explicit (each line names its canto).

## Remaining identity work

In rough order of value:

1. **Run Fix 2 and review.** `make -C 05-registry coref`, read `04-tags/coref.txt`, delete wrong
   proposals, rebuild. This is the human gate — the merge cannot be structurally verified, and
   false positives dominate name-sharing pairs.
2. **Close the 07-relations residual** (finding 2 above) so relation edges pick up merged identities.
3. **Demonstrative / periphrastic labels** (`quel X`, `colui che …`, one-off descriptions): these
   are not figure names and should not be `individual` nodes at all. They need a different treatment
   from Fix 2 (suppression or per-scene typing), not coreference.
4. **Granular invalidation** (optional): a way to clear only the cached scenes a node-set change
   touched, so a rebuild doesn't force a full re-run of the LLM passes. Today this is manual.
